# BuddyUp / v0-ai 产品设计规范

本文档从当前代码库（`app/globals.css`、`components.json`、`components/ui/*` 与业务组件）抽取**可执行**的设计规范，供新页面、新组件与 Code Review 对齐使用。

---

## 1. 设计基调与产品语境

- **视觉气质**：浅色默认；主色与字色偏 **Notion 系**（正文 `#37352f`、侧栏 `#FBFBFA`、列表行悬停/选中为半透明灰叠层）。整体克制、信息密度可高但层级要清晰。
- **技术实现**：**Tailwind CSS v4**（`@import 'tailwindcss'`）+ **`@theme inline`** 将 CSS 变量映射为颜色/字号 token；组件层以 **shadcn/ui（style: new-york, baseColor: neutral）** + **Radix UI** 为基础，图标 **lucide-react**，样式组合使用 **`cn()`**（`clsx` + `tailwind-merge`）。
- **主题**：支持 `next-themes`（`ThemeProvider`）；深色模式变量在 `.dark` 中覆盖。新增语义色时须同时考虑 `.dark`。
- **产品流**（与 UI 一致性相关）：问卷 → 匹配 → 工作台 → 文书等环节应保持**同一套按钮层级、卡片密度、空状态模式**，避免同一概念多种说法（见第 10 节）；**交互节奏与快捷键**见第 7 节；**体验目标与反馈策略**见第 11 节；**气泡引导与 `data-tour` 锚点**见第 12 节。

---

## 2. 单一事实来源（必须遵守）

| 项目 | 路径 | 说明 |
|------|------|------|
| 全局样式与 design token | `app/globals.css` | shadcn `components.json` 中 `tailwind.css` 指向此文件；**以本文件为准** |
| 组件别名 | `components.json` → `@/components/ui` | 新增 shadcn 组件时保持别名一致 |
| 类名合并 | `lib/utils.ts` → `cn()` | 所有条件/覆盖 class 必须用 `cn`，避免 Tailwind 冲突 |

说明：仓库内若存在其他 `globals.css`（例如 `styles/globals.css`），不作为主题源；新增 token 只改 `app/globals.css`。

### 3.4 标签 / 状态扩展色（`:root` 中定义）

以下变量已在主题中定义，用于 **Badge、状态点、轻量标签** 等；使用方式可为 Tailwind 任意值或后续扩展 `@theme`：

- **蓝/橙/红/绿硬标签**：`--tag-blue-bg/text` … `--tag-green-text`
- **软色标签**：`--tag-green-soft-bg/text`、`--tag-blue-soft-text` 等
- **信息/强调 Chip**（已进 `@theme`，可直接用工具类）：  
  - `bg-chip-info-bg` + `text-chip-info-text`  
  - `bg-chip-accent-bg` + `text-chip-accent-text`
- **其它**：`--status-dot-green`、`--success`、`--warning` 等用于状态语义

**规范**：同一语义（成功/警告/错误/信息）全站使用**同一组** token，禁止随意新增 `#` 硬编码色值；若必须新增，同时补齐 `:root` 与 `.dark`，并写入本文档。

### 3.5 图表色

`--chart-1` … `--chart-5` 供 `recharts` 等使用；保持顺序一致以免图例颜色漂移。

## 5. 圆角、边框与阴影

- **全局圆角基准**：`--radius: 0.375rem`（6px）  
  - `radius-sm` = `calc(var(--radius) - 4px)`  
  - `radius-md` = `calc(var(--radius) - 2px)`  
  - `radius-lg` = `var(--radius)`  
  - `radius-xl` = `calc(var(--radius) + 4px)`
- **按钮/输入**：默认 `rounded-md`
- **Card（shadcn）**：`rounded-xl`、`border`、`shadow-sm`、`py-6`、`gap-6`
- **自定义轻卡片**：`.ui-card` → `rounded-xl border border-border/80 bg-background`；内嵌块 `.ui-card-inset` / `.ui-card-inset-soft` 使用 `rounded-lg` + `bg-surface-muted` 系列
- **全局边框**：`@layer base` 中 `* { border-border; outline-ring/50 }`

## 6. 页面与布局

### 6.1 页面壳

- `.ui-page-shell`：`min-h-screen` + 自上而下的极弱渐变（`from-background via-background to-muted/20`），用于整页容器时保持全站一致背景节奏。

### 6.2 列表与表格

- 可交互行：优先 `hover:bg-interactive-hover`、选中 `bg-interactive-active`，与 Notion 行一致。
- 滚动区域隐藏滚动条：工具类 `.hide-scrollbar`（WebKit + Firefox）。

### 6.3 固定/粘性区域

- 避免遮挡主 CTA 或表单提交区；若使用 `fixed`/`sticky` 底栏，需预留 `padding-bottom` 或通过布局留出安全区。

## 7. 交互规范

本节规定**用户能做什么、系统如何响应、何时打断/确认**，与第 8 节组件 API 配合使用。

### 7.1 原则

- **可预测**：同一类控件在全站行为一致（例如主按钮 always 提交当前表单、ghost 为低风险次要操作）。
- **可恢复 / 可感知**：破坏性操作须确认；成功或失败须有明确反馈（Toast、内联错误、`aria-live`），避免「点了没反应」。
- **不打断阅读除非必要**：信息用内联展开/次要面板；仅阻断流程时用模态（`Dialog` / `AlertDialog`）。
- **键盘与指针等价**：自定义可点击区域（卡片、列表行）须具备与 `Button` 等价的键盘操作（见 7.6、9.1）。
- **输入上下文不抢快捷键**：在输入框、富文本、`contenteditable` 内不误触全局快捷键（见 7.2）。

### 7.2 全局快捷键与效率入口

| 快捷键 | 行为 | 实现位置（参考） |
|--------|------|------------------|
| **⌘/Ctrl + B** | 展开/折叠侧栏（桌面）；移动端打开侧栏 Sheet | `components/ui/sidebar.tsx` |
| **⌘/Ctrl + K**（无 Shift） | 打开工作台命令面板；再次按下关闭 | `app/workspace/page.tsx` |

**命令面板（⌘K）约束（须遵守）**：

- 当焦点在 **`INPUT` / `TEXTAREA` / `contenteditable`** 内时，**不**打开命令面板，避免与编辑器快捷键冲突。
- 当事件目标位于 **`[data-skip-workspace-cmdk]`** 包裹区域内时，**不**打开命令面板；嵌入的搜索/命令输入等应加该属性。
- 命令面板内快捷键说明使用 `CommandShortcut`（`text-muted-foreground text-xs tracking-widest`）与主文案对齐。

新增页面级快捷键时：必须 `preventDefault`（若占用浏览器默认）、必须在文档中登记、必须处理「焦点在表单内」的豁免或与产品确认冲突策略。

### 7.3 模态、抽屉与层叠

| 组件 | 适用场景 | 交互约定 |
|------|-----------|----------|
| **Dialog** | 非阻断式补充信息、表单、多步配置 | 打开时焦点移入内容区；关闭按钮带 `sr-only` 文案；`Esc` 由 Radix 关闭（勿重复绑定冲突逻辑） |
| **AlertDialog** | 删除、清空、不可逆操作 | 明确标题与说明；**取消**为安全默认焦点侧；确认动作用 `AlertDialogAction`，destructive 样式与文案一致 |
| **Sheet** | 窄屏侧栏、辅助筛选、移动端侧栏 | 与 Sidebar 移动端模式一致：`SheetContent` 内完整导航结构 |
| **Drawer（vaul）** | 底部抽屉、移动端优先的短流程 |  overlay `bg-black/50`；与 Dialog 不要叠两套全屏遮罩导致无法点击 |

**层叠（z-index）**：常规浮层 `z-50`（shadcn）；引导蒙层（如 `BubbleSpotlightTour`）使用更高 `zIndex`（默认 60），避免被侧栏/顶栏挡住。

### 7.4 导航与上下文保持

- **工作台项目详情**：在当前页内切换详情，**不**无故跳到独立路由丢失左侧列表上下文；返回使用明确「返回」控件（如 `router.back()` + `aria-label="返回"`）。
- **问卷分步**：「上一步 / 下一步」须有 **`aria-label`**（与 `app/questionnaire/page.tsx` 一致）；步骤指示器帮助用户建立位置感。
- **面包屑**：使用 `components/ui/breadcrumb` 时保留 `aria-label="breadcrumb"` 与当前页 `aria-current="page"`。

### 7.5 表单与即时输入

- **提交**：主按钮 `type="submit"` 与表单关联；提交中按钮 `disabled` 或 loading 态，防止双击重复提交。
- **校验**：错误用 `FieldError` / `FormMessage`；控件设 `aria-invalid`；描述区 `aria-describedby` 与 `Form` 组件一致。
- **单行标签内添加（如技能标签）**：`Enter` 触发添加时，在 `onKeyDown` 内处理并避免与表单全局提交冲突（需 `preventDefault` 时明确注释意图）。

### 7.6 可点击卡片与列表项

- 使用 `role="button"` 时，必须 **`tabIndex={0}`**，并在 `onKeyDown` 中响应 **`Enter` 与 `Space`**（`Space` 须 `preventDefault` 以免滚动页面）。
- 卡片内若已有独立按钮（添加/移除），需避免整卡 `onClick` 与内层按钮**事件冒泡**导致误触；内层 `e.stopPropagation()` 或调整命中区域。
- 可选中列表行：悬停 `bg-interactive-hover`、选中 `bg-interactive-active`，与第 6.2 节一致。

### 7.7 拖拽与排序

- 工作台等使用 **@dnd-kit** 的排序：拖拽手柄应可视；键盘替代方案若未实现，须在文案或帮助中说明「当前仅支持指针拖拽」或补齐键盘排序（长期目标）。
- 排序结果持久化（如 `localStorage`）失败时：静默失败须有兜底 UI 或 Toast，避免用户以为已保存。

### 7.8 异步、加载与反馈

- **长时间操作**：按钮 loading（`Spinner`）或骨架屏（`Skeleton`），禁止仅用 disabled 无文案。
- **成功**：轻量操作用 **Sonner** Toast；与主题变量绑定（见 8.8）。
- **失败**：错误信息可读、可行动（重试、检查网络）；表单域旁显示优先于仅 Toast。
- **助手气泡（小布等）**：使用 `aria-live="polite"` 播报新内容，避免打断读屏用户；触发器须有 `aria-label`（见 `workspace-buddy.tsx`）。

### 7.9 引导与蒙层（Spotlight / Tour）

- **Escape**：结束引导并清理状态（`BubbleSpotlightTour`）。
- **焦点**：每步打开时焦点落在**主操作按钮**（`primaryBtnRef`），符合「先读后点」的键盘顺序。
- **装饰性遮罩块**：`aria-hidden`，实际说明与按钮在单独面板内可读。

### 7.10 触控与移动端

- 可点击区域最小建议 **44×44px**（约 `min-h-9 min-w-9` 或更大）；图标按钮使用 `size="icon"` 时注意相邻间距防误触。
- 侧栏在移动端以 **Sheet** 呈现；打开时禁止底层滚动穿透（由 Radix Sheet 处理，勿额外 `overflow:hidden` 破坏焦点）。
- 输入框在移动端使用 `text-base` 直至 `md:text-sm`，避免 iOS 自动缩放（见 4.4）。

### 7.11 空状态与固定栏

- 空状态不仅描述原因，必须提供**主操作入口**（跳转、打开命令面板、去匹配等），见 8.7。
- **底部固定栏**（如匹配页已选数量、去工作台）：不得遮挡页面主 CTA 或最后一题提交区；预留 `padding-bottom` 或使用 `safe-area-inset-bottom`。

### 7.12 破坏性操作与安全网

- **删除 / 移除 / 清空数据**：`AlertDialog` + 说明后果；确认按钮用 `destructive`。
- **仅危险样式**：普通 `Dialog` + `Button variant="destructive"` 可用于非不可逆但需留意的操作，仍建议一句确认文案。

### 7.13 控件视觉与焦点状态（与实现绑定）

- **焦点**：`outline-none` + `focus-visible:ring-ring/50` + `focus-visible:ring-[3px]`；`aria-invalid` 时用 `ring-destructive`、`border-destructive`。
- **禁用**：`disabled:pointer-events-none disabled:opacity-50`；Field 分组用 `group-data-[disabled=true]/field`。
- **悬停**：主按钮 `hover:bg-primary/90`；次级 `hover:bg-secondary/80`；Ghost `hover:bg-accent`；Outline 见深色覆盖。
- **动效**：`tw-animate-css`；打开/关闭短促（约 200ms 量级），不阻塞连续操作。

### 7.14 Tooltip 与延迟

- 侧栏场景 `TooltipProvider delayDuration={0}`；全站新增 Tooltip 时优先 **0 或统一常量**，避免同一屏有的瞬时有的慢。

## 8. 组件使用规范（`components/ui`）

### 8.1 Button

- **variant**：`default`（主）| `secondary` | `outline` | `ghost` | `link` | `destructive`
- **size**：`default`（h-9）| `sm`（h-8）| `lg`（h-10）| `icon` | `icon-sm` | `icon-lg`
- 图标：未指定 `size` 的 SVG 统一 `size-4`；**仅图标的按钮必须 `aria-label`**

### 8.2 Card

- 结构：`Card` → `CardHeader` / `CardTitle` / `CardDescription` / `CardAction` → `CardContent` → `CardFooter`
- 横向留白默认 `px-6`；不要在没有设计理由时改掉 `rounded-xl`/`shadow-sm` 节奏

### 8.3 Field 表单域

- 使用 `FieldSet`、`FieldGroup`、`Field`、`FieldLabel`、`FieldContent`、`FieldDescription`、`FieldError`
- **orientation**：`vertical`（默认）| `horizontal` | `responsive`
- 错误：`FieldError` 为 `role="alert"`、`text-destructive text-sm`
- 描述文字内链接：`text-primary`、`underline`、`underline-offset-4`

### 8.4 Input / Textarea

- 高度与按钮对齐：`h-9`、`rounded-md`、`px-3`
- placeholder：`text-muted-foreground`

### 8.5 Badge

- `rounded-full`、`text-xs`、`border` + variant；可作为链接使用时注意 `[a&]:hover` 已定义

### 8.6 Dialog / Sheet

- 遮罩：`bg-black/50`
- 内容：`rounded-lg`、`border`、`p-6`、`shadow-lg`；关闭按钮右上角，带 `sr-only`「Close」

### 8.7 Empty（空状态）

- 容器：`rounded-lg border-dashed`、`p-6` / `md:p-12`、`text-balance`
- `EmptyTitle`：`text-lg font-medium tracking-tight`
- `EmptyDescription`：`text-sm/relaxed`、`text-muted-foreground`，链接样式与 Field 描述一致
- **规范**：空状态须包含**下一步操作**（主按钮或链接），不能只有说明文案

### 8.8 Toast（Sonner）

- `Toaster` 使用 CSS 变量绑定 `--popover`、`--popover-foreground`、`--border`，与主题同步

### 8.9 Alert

- 默认/破坏性两 variant；带图标时网格对齐；描述区 `text-muted-foreground`（destructive  variant 下描述色略调）

### 8.10 Tooltip

- 侧栏 Provider 中 `delayDuration={0}`；全站 Tooltip 行为应一致，避免有的瞬时有的延迟过长

## 10. 文案与信息架构

- **简短、动作导向**：按钮用动词；避免冗长营销句。
- **同一概念同一用词**：例如「项目清单」「问卷」「匹配」「工作台」等与产品流一致，不混用英文/中文别名。
- **数字与元信息**：列表卡片中指标（截止日期、费用等）保持**同一顺序与图标**（见 ProgramCard 中 Clock / DollarSign 等模式）。
- **图标-only**：必须配合 `aria-label` 或 Tooltip + 可见文本；装饰性图标 `aria-hidden`。

## 11. 用户体验规范

本节约定**用户从进入到完成目标**过程中的心理模型、反馈与容错，与第 7 节（行为）、第 8 节（组件）配合。

### 11.1 体验目标

- **本地优先、隐私可感知**：当前产品为游客/本地存储形态时，须在首屏或常驻区域让用户理解「数据在本地」（见 `GuestBanner`：**游客模式 · 数据仅本地保存**）；避免用户误以为已云端同步。
- **任务可完成**：每一主流程（问卷填完 → 出匹配 → 加项目 → 工作台管状态/文书）在 UI 上始终有**下一步**，不出现「死胡同」仅说明无操作。
- **控制感**：长耗时（匹配生成、AI 类操作）须展示**阶段或进度**（如「分析进度」与分步列表），而非无限白屏或单一转圈无文案。
- **可恢复**：误操作优先用撤销 Toast、二次确认、或「返回上一步」补救；不可逆操作走 `AlertDialog`（见第 7.12 节）。

### 11.2 心智模型与信息层级

- **一处一义**：同一指标或状态不在一屏重复堆叠（与 design-spec-governance 一致）；若需总览 + 明细，总览用压缩数字，明细在展开区或二级视图。
- **主次分明**：每屏**一个主 CTA**（主按钮 `default`）；次要操作用 `outline` / `ghost` / `secondary`，避免多个实心主色按钮竞争。
- **进度可见**：问卷使用步骤指示（桌面横向步骤条 + 移动端「步骤 x / n」），当前步高亮、已完成可区分；**必填**用 `*` 与文案一致标注。

### 11.3 加载、骨架与首屏

- **全页等待**：数据未就绪时可用**居中转圈** + `bg-background` 或 `.ui-page-shell` 系背景，避免布局跳动。
- **块内加载**：优先 **Skeleton** 占位列表/卡片，与真实内容布局接近。
- **长任务**：除转圈外须有**标题 + 一段说明**（见匹配页生成态：主标题 + `text-muted-foreground` 说明 + 进度区块），降低焦虑。

### 11.4 成功、错误与空状态

- **成功**：轻量操作用 **Toast**；关键结果（如保存成功）若用户可能离开页面，可 Toast + 界面内状态同步。
- **错误**：表单用 `FieldError`；全局或网络错误用 **Alert** 或 Dialog 内说明，并给出**下一步**（重试、返回、联系支持占位）。
- **空列表**：使用 `Empty` 系列（第 8.7 节）：标题 + 说明 + **主操作**（去匹配、打开命令面板、添加项目等），禁止仅「暂无数据」四字。

### 11.5 渐进披露与密度

- **默认可扫读**：卡片首屏展示标题、关键元数据、主操作；展开区收纳次要字段（如 ProgramCard 展开详情）。
- **折叠/展开**：同一控件「展开 / 收起」文案与图标状态一致；动画短，不拖慢重复操作。

### 11.6 一致性与术语

- 与第 10 节一致：产品流名词（问卷、匹配、工作台、项目清单、文书等）**全站统一**；状态词（待申请、准备中、已提交等）与筛选器、标签同色同义。

### 11.7 测试与演示入口（不干扰正式用户）

- **全局测试菜单**（开发/内测）：须有明确 `aria-label`（如「测试：跳转界面与演示」），样式上不过分抢眼但可被需要的人发现。
- **重播引导**：通过 URL 参数或菜单清除对应 `localStorage` 后重播（如匹配页 `replaySpotlight`、工作台内测重播逻辑），避免正式用户反复被强制引导。

## 12. 显示引导规范

本节约定**首次使用引导、高亮锚点、文案与频率**，实现上对齐 `BubbleSpotlightTour` 与各页 `data-tour` 约定。

### 12.1 组件与能力（`BubbleSpotlightTour`）

- **步骤结构**：每步包含 `targetSelector`（CSS 选择器）、`title`、`description`；末步可设 `finishButtonLabel`（默认「完成」类语义由实现决定）。
- **高亮**：通过 `readRect` 计算目标元素视口矩形；`padding` 控制挖孔外扩像素。
- **目标缺失**：`allowMissingTarget: true` 时，无有效矩形则**整屏压暗 + 仅气泡文案**，用于说明「尚未出现」的 UI（如尚无项目时的列表区、底部栏）；文案必须解释**前提条件**与**用户可做的事**。
- **持久化**：完成或关闭时写入 `finishStorageKey` 对应 **localStorage**（值为 `"1"`）；键名集中在 `lib/onboarding-keys.ts` / `lib/buddyup-local-storage.ts`，禁止页面内硬编码散落字符串。
- **键盘与焦点**：打开时焦点进主按钮；`Escape` 结束并 `finish`（与第 7.9 节一致）。
- **布局跟随**：监听 `scroll`（capture）与 `resize` 更新挖孔位置，避免滚动后高亮错位。

### 12.2 锚点约定（`data-tour`）

被引导元素应设置稳定 **`data-tour="<kebab-case-id>"`**，选择器写作 `'[data-tour="..."]'`，避免依赖易变的 class。

| `data-tour` 值（示例） | 含义 |
|------------------------|------|
| `match-category-filter` | 匹配页分档筛选 |
| `match-school-list` | 匹配页院校列表 |
| `match-program-toolbar` | 项目列表工具栏 / 批量勾选区 |
| `match-program-add` | 单卡「加入清单」操作（首卡可通过 `tourAddTarget` 打标） |
| `match-bottom-cta` | 底部进入工作台栏 |
| `workspace-sidebar-shell` | 工作台左侧栏容器 |
| `workspace-buddy-anchor` | 小布助手锚点 |

**新增引导步**时：必须新增或复用 `data-tour`，并在本节表格登记；若锚点在子组件内，通过 props（如 `tourAddTarget`）在**唯一实例**上输出 `data-tour`，避免多元素同 id。

### 12.3 匹配页引导（产品逻辑摘要）

- **触发**：首次进入匹配结果且本地无 `buddyup_onboarding_match_spotlight_v1` 时自动打开；`?replaySpotlight=1` 可清键重播。
- **步骤顺序**：分档筛选 → 院校列表 → 项目工具栏 →（有项目时）加入清单 → 底部进入工作台（`allowMissingTarget` 适配无底部栏时仍可读说明）。
- **动态文案**：底部栏一步的 `description` 随「是否已选项目」变化，避免与真实 UI 矛盾。

### 12.4 工作台引导（产品逻辑摘要）

- **触发条件**（与实现对齐）：草稿与问卷加载完成、满足桌面 Spotlight 条件、当前为 dashboard 主视图、**未**选中学校/项目、且本地无 `buddyup_onboarding_workspace_spotlight_v1`。
- **步骤**：左侧栏（含 ⌘K 说明）→ 申请列表（无项目时文案引导回匹配）→ 小布（`finishButtonLabel` 可为「我知道了」）。
- **重播**：测试菜单或 `openWorkspaceSpotlightReplay` 类逻辑清除 `ONBOARDING_WORKSPACE_SPOTLIGHT_V1` 后 `requestAnimationFrame` 打开。

### 12.5 文案写作（引导专用）

- **title**：短名词或动宾短语（≤ 约 12 字为宜），点明「这是什么」。
- **description**：1～3 句，先**价值**再**操作**；提及快捷键时用 **⌘/Ctrl** 并列，适配 Mac 与 Windows。
- **语气**：帮助式、不责备；不用「你应该」类说教，用「可以」「在此」等中性指引。

### 12.6 频率与打扰控制

- **每种引导对应唯一 storage key**；用户完成后不再自动弹出同版本引导。
- **版本升级**：若引导流程大改，应使用**新 key**（如 `_v2`）而非覆盖旧行为，避免老用户无法看到新说明或新用户被旧标记跳过。
- **不要叠引导**：同一会话避免多个全屏蒙层队列；进入页面前先决定优先级（产品引导 > 次要提示）。

### 12.7 无障碍与视觉

- 遮罩大块使用 `aria-hidden`；步骤文案与按钮在气泡面板内可被读屏读取。
- 挖孔区域对比度：浅色主题 overlay 使用 `bg-foreground/45` 等（与实现一致），保证气泡内文字与按钮仍清晰（见 `bubble-spotlight-tour.tsx`）。

## 13. 无障碍基线（与仓库技能对齐）

- 图标按钮：`aria-label`
- 状态不只靠颜色：配合文案、图标或 `aria-*`
- 主路径可键盘完成：焦点顺序合理、`focus-visible` 可见
- 表单：label 关联、`aria-invalid`、错误信息可Announcement（`FieldError`）
- 对比度：主文与背景、链接与背景在浅色主题下保持可读；深色主题变更后需抽样检查
