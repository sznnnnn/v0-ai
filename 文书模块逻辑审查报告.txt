文书模块逻辑审查报告（当前版本）
审查范围：
1) app/workspace/write/[programId]/page.tsx
2) app/workspace/write/[programId]/materials/page.tsx
3) app/workspace/write/[programId]/signals/page.tsx
4) app/workspace/write/[programId]/cards/page.tsx
5) app/workspace/write/[programId]/questions/[step]/page.tsx
6) app/workspace/page.tsx 与 app/workspace/program/[programId]/page.tsx 的文书入口
7) lib/document-drafts.ts

==================================================
一、P0（高优先级，影响正确性/数据一致性）
==================================================

P0-1 中英文双稿可能不一致（切换到英文优先展示旧缓存）
现象：
- 用户修改了中文正文后，点击语言切换按钮进入英文。
- 如果 enContent 已有内容，系统直接显示旧英文，不会重新翻译当前中文。
影响：
- 中英文版本发生“静默分叉”，用户误以为是同一版本的双语稿。
代码线索：
- write/[programId]/page.tsx 中 switchLanguage() 对 enContent.trim() 的直接返回逻辑。
建议：
- 增加“源内容哈希”或“上次翻译来源时间戳”比对。
- 当中文已变更且英文过期时，提示“英文需更新翻译”。

P0-2 应用建议（Accept）存在并发覆盖风险
现象：
- 点击 Accept 后有 520ms 延迟再执行替换。
- 若用户在这段时间继续输入，可能被旧闭包内容覆盖。
影响：
- 用户最新编辑被回滚，属于严重编辑器体验问题。
代码线索：
- write/[programId]/page.tsx applyCritiqueSuggestion() 内 setTimeout + 使用 content 快照。
建议：
- 使用函数式 setState 或 ref 持有最新 content。
- 应用建议期间锁定编辑区，或在提交前做冲突检查（diff/版本号）。

P0-3 标注区间 range 计算与实际正文索引可能偏移
现象：
- buildCritiqueRanges() 用 text.trim() 计算 start/end。
- 后续替换使用原始 content 的 index。
影响：
- 前后空白/换行存在时，可能高亮错位或替换到错误位置。
代码线索：
- lib 内/页面内 buildCritiqueRanges 的 trim 使用。
建议：
- range 统一基于原始 content，不做 trim。
- 仅在“是否为空”判断时用 trim。

==================================================
二、P1（中优先级，影响交互闭环/流程一致性）
==================================================

P1-1 “写文书”入口弹窗逻辑未全覆盖全部入口
现象：
- 已覆盖：项目详情页与工作台详情区“写文书”按钮。
- 未覆盖：侧栏“文书草稿”Link、已保存文书卡片Link、命令面板里文书草稿直跳。
影响：
- 用户在某些入口仍绕过“打开旧稿/创建新稿”分流，行为不一致。
建议：
- 抽一个统一的 openWriteFlow(programId) 工具。
- 所有导航入口统一调用该函数，而非直接 href/router.push。

P1-2 questions 路由已改为重定向，但旧步骤页仍保留
现象：
- /write/[programId]/questions/[step] 会自动 replace 到编辑页。
影响：
- 逻辑上可用，但残留“僵尸路由”，后续维护易混淆。
建议：
- 若确认废弃，清理 materials/signals/cards/questions 四个旧流程页或明确标记 legacy。

P1-3 批注状态与重扫逻辑存在概念混用
现象：
- 卡片列表只展示 open，resolved/dismissed 会消失。
- 仍保留统计、重扫与反馈文案，但头部块已删除后可见性下降。
影响：
- 用户处理后“去哪里看历史处理项”不明确。
建议：
- 增加“已处理”折叠区，或提供“显示全部”开关。
- 把重扫入口固定到右侧底部或工具栏，保证可发现。

==================================================
三、P2（低优先级，影响可维护性/性能）
==================================================

P2-1 演示逻辑与生产逻辑耦合较重
现象：
- Demo critiques、demo draft 生成、动效状态、应用逻辑都在单一 page.tsx。
影响：
- 文件复杂度高，后续改动风险大。
建议：
- 拆分为 hooks + ui components：
  - useCritiqueEngine
  - useBilingualDraft
  - CritiqueSidebar
  - AnnotatedDraftView

P2-2 range 重扫频率与动画状态可能互相干扰
现象：
- 内容每次变化 180ms 去抖重扫 range。
- 同时有 applying 动效与退出动画。
影响：
- 极端快速输入下，可能出现高亮闪动。
建议：
- 引入“应用建议事务锁”（applying 时暂停自动重扫）。
- 动画结束后再统一重算。

P2-3 “创建新的”是全量清空，缺少撤销保护
现象：
- clearProgramDrafts 会删除该项目全部 kind 草稿与版本。
影响：
- 误点后不可恢复。
建议：
- 二次确认（危险动作）或提供“仅新建当前 PS”选项。
- 可选：软删除回收站（本地保存 N 分钟）。

==================================================
四、流程侧漏考虑（产品逻辑还可补）
==================================================

1) 双语切换缺少“版本来源提示”
- 建议在英文模式顶部显示“基于中文版本于 XX:XX 翻译”。

2) 高亮“好句/问题句”缺少冲突判定
- 同一句可能同时满足“好句”和“问题句”规则，当前未定义优先级策略。

3) Accept 后占位符校验缺失
- 目前会插入【】占位模板，但没有提示“仍有占位未填”。
- 建议提交前校验：检测【...】残留并提示。

4) 文书入口分流策略未按 kind 维度区分
- 当前“是否已有草稿”按全部 kind 判断。
- 可能出现：只有 cv 有稿，点击“写文书(PS)”也弹窗。
- 建议按当前入口目标 kind（默认 ps）判断。

结论
==================================================

当前文书模块已经具备“可演示闭环”，但在真实用户高频编辑场景下，仍有 3 个高优先级一致性问题（双语版本一致性、Accept 并发覆盖、range 索引偏移）需要优先修复。完成这三项后，稳定性会有明显提升。
